# Data Agent Backend: краткое резюме для руководителя

Дата обзора: 2026-06-01

## Общий вывод

Backend уже решает основную продуктовую задачу: принимает данные, анализирует их через AI/SQL, помогает строить дашборды и умеет экспортировать результат во внешние BI-системы. Это не прототип "на бумаге", а рабочий сервис с большой доменной логикой.

Но по качеству это скорее сильный MVP, чем production-ready система. Основной риск сейчас не в том, что продукт не работает, а в том, что при реальной эксплуатации могут проявиться проблемы безопасности, стабильности и поддержки.

Короткая оценка:

- Функциональность: высокая.
- Архитектура для MVP: нормальная.
- Готовность к production: требует доработки.
- Главный фокус ближайших работ: безопасность, доступы, стабильность, тесты.

## Что в проекте уже хорошо

- Есть полноценный backend на FastAPI.
- Есть авторизация, пользователи, роли и квоты.
- Есть хранение пользовательских сессий и истории.
- Есть работа с CSV/Excel и аналитика поверх данных.
- Есть AI-чат, SQL-ответы и генерация дашбордов.
- Есть анализ изображений/скриншотов дашбордов.
- Есть экспорт в BI-системы: DataLens, Visiology, Foresight, Navigator.
- Есть Docker и документация для запуска.

То есть основная бизнес-ценность уже собрана. Проект не нужно переписывать с нуля.

## Основные риски простыми словами

### 1. В коде есть секреты и пароли

В репозитории обнаружены пароли, client secret и тестовые учётки. Даже если часть из них demo, это плохая практика и риск безопасности.

Что нужно сделать:

- Убрать секреты из кода.
- Перенести их в переменные окружения или секреты CI/CD.
- Поменять реальные пароли и ключи, если они могли попасть в git.

### 2. Есть потенциально опасные SQL-места

Часть API позволяет выполнять SQL или подставлять имена таблиц недостаточно безопасно. Для внутреннего MVP это могло быть удобно, но для production это риск.

Что нужно сделать:

- Закрыть прямой SQL endpoint или разрешить его только администраторам.
- Добавить безопасную валидацию имён таблиц.
- Запретить опасные операции на уровне backend.

### 3. Внешние BI-интеграции доступны слишком широко

Сейчас экспорт и публикация во внешние BI-системы защищены фактом логина, но не отдельной ролью. Это означает, что обычный пользователь потенциально может запускать действия, которые должны быть доступны только администратору или интегратору.

Что нужно сделать:

- Ввести отдельную роль для публикации/интеграций.
- Или временно закрыть export-функции только для admin.

### 4. OIDC/SSO нужно усилить

SSO уже реализован, но в нём есть упрощения: отключена строгая TLS-проверка и не проверяется `state` в login flow.

Что нужно сделать:

- Включить TLS verification.
- Добавить проверку `state`.
- Проверять токены по стандартному OIDC flow.

### 5. Есть риск нестабильности под нагрузкой

DuckDB-соединения открываются, но не закрываются достаточно явно. Vision-анализ и Foresight-интеграция выполняют тяжёлые операции внутри backend-процесса.

Что нужно сделать:

- Исправить жизненный цикл DuckDB-соединений.
- Тяжёлые задачи постепенно вынести в фоновые jobs.
- Хранить статус jobs в базе, а не только в памяти процесса.

### 6. Тестов и CI пока недостаточно

Есть ручные e2e/probe-скрипты, но нет нормального набора автотестов и CI, который каждый раз проверяет backend.

Что нужно сделать:

- Добавить минимальные автотесты.
- Добавить lint/type checks.
- Добавить CI на тесты и сборку Docker image.

## Что реально сделать в первую очередь

### Срочный этап: безопасность

Срок: примерно 1-2 дня.

Задачи:

- Убрать секреты из кода.
- Закрыть прямой SQL endpoint.
- Добавить безопасную обработку имён таблиц.
- Ограничить export-интеграции ролью admin/publisher.
- Проверить production-настройки cookie и JWT.

Результат:

- Снижается риск компрометации и опасных действий через API.

### Второй этап: SSO и доступы

Срок: примерно 1-2 дня.

Задачи:

- Усилить OIDC login flow.
- Включить TLS verification.
- Добавить проверку `state`.
- Продумать роли: user, admin, publisher/integrator.

Результат:

- Авторизация становится ближе к production-уровню.

### Третий этап: стабильность

Срок: примерно 2-4 дня.

Задачи:

- Починить DuckDB-соединения.
- Убрать лишний SPA fallback из backend или поправить health endpoint.
- Не отдавать внутренние ошибки клиенту.
- Добавить базовый набор автотестов.

Результат:

- Backend становится стабильнее и предсказуемее.

### Четвёртый этап: поддерживаемость

Срок: примерно 1-2 недели.

Задачи:

- Ввести миграции БД.
- Вынести тяжёлые vision/Foresight задачи в background jobs.
- Разделить очень большие файлы на понятные модули.
- Унифицировать общие helpers для схем, типов графиков и JSON от LLM.
- Добавить CI.

Результат:

- Команде проще развивать проект без регрессий.

## Приоритеты по важности

1. Безопасность: секреты, SQL, роли.
2. Auth/SSO: корректный OIDC и cookie/JWT настройки.
3. Стабильность: DuckDB connections и тяжёлые jobs.
4. Тесты и CI: минимальная автоматическая проверка.
5. Рефакторинг: большие файлы и дублирование.

## Итоговая рекомендация

Проект стоит не переписывать, а стабилизировать. В нём уже много готовой бизнес-логики, особенно вокруг AI, дашбордов и BI-интеграций.

Оптимальный подход: короткий hardening-спринт на безопасность и стабильность, затем аккуратное улучшение архитектуры без остановки продуктовой разработки.

Рекомендуемый ближайший план:

- Неделя 1: закрыть критичные security issues и доступы.
- Неделя 2: стабилизировать ресурсы, добавить тесты и CI.
- Далее: постепенно выносить тяжёлые jobs и упрощать самые большие модули.
